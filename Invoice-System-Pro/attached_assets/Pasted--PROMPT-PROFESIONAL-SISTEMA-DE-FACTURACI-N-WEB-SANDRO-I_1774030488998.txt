7. Encabezado con logo siempre visible en impresión
8. El número de factura debe generarse **automáticamente** y ser único
9. Ocultar en impresión: botones, navegación, sidebar, cualquier elemento de UI

## ⚙️ ARQUITECTURA BACKEND — REQUISITOS DE ESCALABILIDAD

### Base de datos (PostgreSQL con Prisma):
```prisma
model Invoice {
  id          Int       @id @default(autoincrement())
  invoiceNumber String  @unique
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
  deletedAt   DateTime? // soft delete
  clientId    Int
  client      Client    @relation(fields: [clientId], references: [id])
  measuredById Int?
  quotedById   Int?
  budgetDate  DateTime?
  paymentMethod String  // "50%" | "100%" | "contra entrega"
  downPayment Decimal   @db.Decimal(12,2)
  notes       String?
  status      InvoiceStatus @default(PENDING)
  items       InvoiceItem[]
  total       Decimal   @db.Decimal(12,2)
  userId      Int       // quién creó la factura
}

model InvoiceItem {
  id          Int     @id @default(autoincrement())
  invoiceId   Int
  invoice     Invoice @relation(fields: [invoiceId], references: [id], onDelete: Cascade)
  productId   Int?
  product     Product? @relation(fields: [productId], references: [id])
  description String
  quantity    Decimal @db.Decimal(10,2)
  unitPrice   Decimal @db.Decimal(12,2)
  total       Decimal @db.Decimal(12,2)
}
```

### API REST con buenas prácticas:
- Paginación obligatoria en todos los listados (`?page=1&limit=20`)
- Índices en columnas de búsqueda frecuente (clientId, invoiceNumber, createdAt)
- Rate limiting por usuario (express-rate-limit)
- Validación de inputs con Zod o Joi
- Manejo centralizado de errores con middlewares
- Logs estructurados (winston o pino)
- Variables de entorno para todos los secretos (nunca hardcoded)
- CORS configurado correctamente para producción
- Transacciones de DB para operaciones que modifican múltiples tablas

### Concurrencia y consistencia:
- El número de factura debe generarse con un **sequence de PostgreSQL** para garantizar unicidad sin race conditions
- No usar MAX(id)+1 para generar números de factura
- Usar `SELECT FOR UPDATE` o transacciones donde sea necesario

## 📐 REGLAS DE DISEÑO FRONTEND (NO NEGOCIABLES)

1. **Responsive**: funcional en desktop (1280px+) y tablet (768px+). Mobile es secundario pero no debe romperse.
2. **Sidebar fijo** con íconos + texto: Dashboard, Facturas, Nuevo Presupuesto, Clientes, Productos, Personal, Configuración
3. **Los contenedores de datos NUNCA deben desbordarse**: usar `overflow-x: auto` en tablas, `text-overflow: ellipsis` en celdas con texto largo, tooltips al hacer hover
4. **Tipografía**: usar Google Fonts — `'Montserrat'` para títulos, `'Source Sans 3'` para cuerpo
5. **Sin errores de márgenes**: todos los formularios deben tener padding consistente, las facturas impresas deben respetar los márgenes del papel
6. **Estado vacío**: cuando no hay datos, mostrar ilustración SVG simple con CTA ("Crear primera factura")
7. **Feedback visual**: loading skeletons durante carga, toast notifications para éxito/error, confirmación modal para eliminaciones
8. **Accesibilidad básica**: labels en inputs, contraste mínimo AA, foco visible con teclado
